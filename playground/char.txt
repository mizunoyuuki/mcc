Cではcharは整数型であり、文字型でもある。

char c = 65;
char c = 'A'; // 全く同じ。'A'は単なる整数定数 65

'A'というリテラルは C では int型の整数定数(値65)である。
文字としての特別な型は存在しない。

---
整数としてのchar

signed char     -128 - 127
unsigned char   0 - 255
char            どちらか (処理系依存)

[255まで]ってのはunsigned charのこと。
char は処理系依存

「文字」として扱われるのは表示・解釈の時

char c = 65;
printf("%d", c);   // -> 65
printf("%c", c);   // -> A

中身は同じ整数だが、printf
のフォーマット指定子が「この値をASCII文字」として解釈して表示する

## Cで'A'はcharではなくintである理由

C89の規格で文字リテラル'A'の型はintと定められている。(C++ではchar)
これはken thonmsonとdennis ritchieがB言語からC言語を設計した際、
文字も整数もワードサイズで扱う。という方針を引き継いだため。

sizeof('A')をコンパイルすると4(intのサイズ)になる。
C++では1が返る。

この違いで、C->C++への移行でバグが生まれることがある。


- unsigned charがC言語で特別扱いされる理由

Cの規格でunsigned charは「オブジェクトのバイト表現にアクセスする型」として特別な地位を持つ
任意のポインタをunsigned char* にキャストしてバイト単位でメモリを読むことが規格上許可されており
(strict aliasingの例外), memcpyやmemsetの実装がこれを利用している。
intのバイト表現をunsigned char* で読んでエンディアンを調べるコードはCでは合法ですが、
int *をfloat*で読むのは未定義動作。

C言語では、charやshortを演算に使うと、自動的にintに昇格する。
これは、1970年代のPDP-11などの初期ターゲットCPUがワード(16bit/32bit)単位で演算する方が効率的だったために設計されたルール。
文字定数が最初からintなのはも、「どうせpromoteされるなら最初っからintでいい」という設計しそうの延長にあるみたい。


char c = 'A';
=> 'A'(int)からcharへ暗黙変換して格納する

promotionが起きるのは演算の瞬間だけ。
