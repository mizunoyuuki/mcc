## Cの「前方宣言(forward declaration)」が必要になるのは、Cコンパイラがワンパス(一回走査)で処理する設計だから。

Cはコンパイラが入力を先頭から末尾まで一回だけ読んで処理する「ワンパスコンパイル」を前提に設計されている。
そのため、使用する型や関数は使用箇所より前に宣言されている必要がある。これは昔のメモリ制約に起因する設計。

構造体のMemberにオフセットを持たせる設計は、コンパイラ実装において「構造体のレイアウトを型定義時に確定させる」という重要な設計判断。

つまり、実行時にメンバのオフセットを計算する必要がない。実行時のオーバーヘッドがゼロ。
このコンパイル時に確定する性質のおかげで、Member構造体にint offsetを持たせるだけでいい。

一方、PythonやJavascriptのオブジェクトは、メンバのレイアウトが動的に変わるため、メンバアクセスにハッシュテーブル検索が必要で、Cの構造体アクセスより数十倍遅くなる。
V8のhidden class最適化は、このCの「コンパイル時レイアウト確定」をJITで模倣する技術である。

Type構造体に追加したint align
「この型の値をメモリに配置する時、アドレスが何の倍数でなければならないか」を表す値。

struct S {
  char a; // size=1, align=1 -> offset 0
  int b;  // size=4, allign=4 ->offset ???
};

aがoffset 0で 1バイトを使うので、次は offset 1 から空いています。しかしintのalignは4なので、offsetは1におけない。
次の4の倍数 -> offset 4に配置する。

豆知識:
x86-64はアラインされていないメモリアクセスをハードウェアが自動制御するため「動いてしまう」が、パフォーマンスペナルティがある
x8664アーキテクチャは、歴史的にアンアラインドアクセスを許容する設計になっており、4の倍数ではないアドレスからintを読み書きしてもクラッシュはしない。
しかし内部的にはキャッシュラインをまたぐアクセスが発生し、intelの最適化マニュアルによると2-3倍の遅延が生じることがある。

特にSIMD命令(SSE/AVX)では16バイトや32バイトのアライメントが要求され、アンアラインドアクセスは #GP (General Protection Fault)で即クラッシュする命令もある。
そのため「x86だからalignは気にしなくていい」は誤りで、正しいアラインメントは性能と、ABI準拠の両面で重要。

ARMアーキテクチャ(特にARMv5以前)では、アンアラインドアクセスがハードウェア例外を引き起こし、即座にプログラムがクラッシュする。
