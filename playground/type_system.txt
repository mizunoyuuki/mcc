コンパイラの「パス」の概念は、歴史的にメモリ制約から生まれた
初期のコンパイラ「1950-1960」とかはメモリが限られていたため、ソースコードを一度にメモリに載せることができなかった。そのため、入力を複数回読み直す「マルチパス」方式が採用されていた。　書くパスが特定の処理（字句解析、構文解析、意味解析、コード生成）を担当し、中間結果をファイルに書き出して次のパスに渡すという形式。
現代のコンパイラでは、メモリが潤沢なため、AST上で複数の操作を行う形になってるが、「パス」はそういったメモリ制約から生まれたものだった。

型チェックには、「構造的型付け」と「名前的型付け」という二つの大きな流派がある。
構造的型付けは、型の内部構造（フィールドやメソッドの一致）から互換性を判断し、名前的型付けは、型の名前・宣言かで判断する。
C言語は、基本的に名前的型付けですが、struct同士の代入互換性などでは構造も考慮される。

TypeScriptは構造的型付けの代表例で、同じ形のオブジェクトなら、異なる型名でも互換とみなされる。
この違いは、意味解析パスの実装に大きく異なる


###型システムの実装。

型は、変数にだけつけて管理しておけばいいものではない。(型が決定するのは、変数などのprimaryの要素である。)

全ノードに型をつける仕組み。
基本的な考え方は、ボトムアップの型伝播。
ASTは木構造なので、葉ノードから根に向かって型を決定していく。

      ND_ADD  ← int（子が両方intだから）
      /    \
   ND_LVAR  ND_NUM  ← それぞれ既に型がわかる
    (int)    (int)


int a = 10 + 10;   => ⚪︎
int* a = 10 + 10;  => ×
こういったのを把握できるように。

###この実装とは何か。

1. 型の付与/型付け: Type Annotation

2. 意味解析: (semantics analysis)
- 未定義変数の検出
- 二重定義チェック(parse_declarationでやってる)
- スコープの解決

3. 型システム
これはちょっと上位の概念。
言語設計レベル「型のルールの体系」
「int + int = int」
「pointer + int = pointer」
「pointer同士の掛け算は無理」

add_typeはそのルールを実装したもの。

- 「意味解析」という用語は自然言語処理から借用された概念
コンパイラ理論での「意味解析(semantic analysis)」は、もともと言語学の分野で使われていた用語
言語学者Noam Chomskyの有名な著書[Syntactic Structures]で構文(syntacx)と意味(semantic)を明確に分けたことが影響している。
Chomskyの有名な例文「Coloress green ideas sleep furiously(無職の緑の考えが猛烈に眠る)」
は、構文的には正しいが意味的にはナンセンスな文として、構文と意味の違いを示した。

コンパイルにおけるint a = "hello";  も、構文的には正しいが意味的には間違っている。

- 型システムの健全性(Soundness)は
型システムが「健全」であるとは、「型検査を通過したプログラムは、実行時に型エラーを起こさない」という保証のこと
しかしC言語は、キャストやvoidポインタによって型安全を破ることが許されている。
C言語は「プログラマを信頼する」という設計哲学を持っているためで、安全性より柔軟性を優先した結果。

- 「型付け」と「型推論」は似ているが、厳密には異なる概念。
「型付け」はの＝どに型を割り当てる行為全般を指す。
「型推論(type inference)」は、「プログラマが明示的に型を書かなくても、コンパイラが自動的に型を導出する」
C言語の変数の型付けは、「型推論」ではなく。「型検査(type checking)」に近い。
一方、add_typeで、ND_ADDノードの型を「左辺と右辺から決定する」の部分は、型推論に近い。


各ノードごとに型をつけるルールは違う。
ND_NUM                : 常にint
ND_LVAR               : 既にlvar->typeから取れている。
ND_ADD, ND_SUB,       : lhs, rhsの型の型を見る。両方intの場合は、int, 片方がptrの場合は、ptr
ND_MUL, ND_DIV        : 常にint (ptr同士の乗除はないため)
ND_EQ, ND_NE, ND_LT等 : 常にint (比較結果真偽値=整数)
ND_ADSSIGN            : 左辺の型をそのまま採用
ND_ADDR(&x)           : 子の型を「一段ポインタで進む」子がintならint*
ND_DEREF(*x)          : 子の型を「一段剥がす」子がint*ならint
ND_FUNCALL            : 今はint固定、将来的には、関数の戻り値型を参照
