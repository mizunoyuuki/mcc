配列からポインタへの暗黙の型変換を実装する。

配列とポインタはよく組み合わせて使われるので、Cでは構文上、ポインタと配列をあまり区別せずとも、なんとなく動くようになっている。
それが裏目に出て、配列とポインタの関係がどうなっているのか、プログラマにとっては分かりにくくなっている。
Cにおいて、配列とポインタは完全に別の型です。

ポインタは(x8664では)8バイトの値の型です。
intに対して+や-といった演算子が定義されているように、ポインタに対しても+や-が定義されています。
ポインタはそれに加えて、単項*演算子が定義されていて、それを使うことでポインタの指している先を参照することができる。
単項*を除けば、ポインタはそれほど特別なことはない。


一方で配列は何バイトにでもなりうる型。ポインタとは異なり、配列に対しては演算子がほとんど定義されていません。
定義されている演算子は、配列のサイズを返すsizeof演算子と、配列の先頭の要素のポインタを返す&だけ。
それ以外、配列に対して適用できる演算子はない。

ではなぜ、a[3]といった式がコンパイルできるのか。
Cでは、a[3]は、*(a-3)と等価であるものとして定義されている。。。。。
配列に対して+演算子はなかったのではないのですか？

ここで配列が暗黙のうちにポインタに変換される。と言う文法が効いてくる。
sizeofか単項&のオペランドとして使われる時以外は、配列は、その配列の先頭要素を指すポインタに暗黙のうちに変換されることになっている。
したがって、*(a + 3)は、配列aの先頭要素を指すポインタに3を足したものをでリファレンスする。と言う式になり、それは結果的に配列の3番目の要素をアクセスするのと同じ意味になる。

Cにおいては、配列アクセスのための[]演算子と言うものはありません。
Cの[]は、ポインタ経由で配列の要素にアクセスするための簡便な記法にすぎない。

同様に、関数引数として配列を渡すと、その配列の先頭要素へのポインタになったり、ポインタに対して配列を直接代入しているかのような書き方ができたりする。

と言うわけで、コンパイラはほとんどの演算子の実装において、配列をポインタに型変換すると言うことを行わなければいけない。
sizeofと&以外の場合は、演算子のオペランドをパーズしたら、その型がTの配列だったらTへのポインタにしてしまう。とすればいい。

ここまで、完成すれば次のようなコードが動くようになる。

int a[2];
*a = 1;
*(a + 1) = 2;
int *p;
p = a;
return *p + *(p + 1); // -> 3


## 豆知識
ちなみに、1978年のK&R初版の時点で、配列がほとんどの文脈でポインタに変換されるルールが存在していた。
これは元々のB言語の設計に起因する。
B言語では、配列は「先頭要素へのポインタを格納するメモリセル」として実装されていた。
つまり、int a[10];は、内部的に、「ポインタ変数a + 10個のint領域」で11個分のメモリを使っていた。
デニスリッチーは、Cを設計する際に、この世分なポインタ変数を廃止して配列を、「連続したメモリ領域そのもの」に変えたが、既存のB言語コードとの互換性のために「式中では配列がポインタに化ける」と言うdecayルールとした。

この歴史的経緯は、「The Development of the C Language」と言う論文で解説されている。

- decayが起きない三つの例外がある。
C11規格では、配列がポインタにdecayしない場面は正確には３つ定義されている。

1. sizeofのオペランド
2. 単項&オペランド
3. 文字列リテラルによる配列の初期化 (char s[] = "hello\n")
。

- 
int *p;
p = a;
でポインタ変数に配列を代入できる。のは、代入式で配列がポインタにdecayするから。
これは「配列をポインタに代入している」のではなく、「decayで生成されたポインタ値をポインタ変数に代入している」が正確な解釈になる。
逆の
int a[10];
a = p;
はコンパイルエラーになる。
配列は「変更可能な左辺値(modifiable lvalue)」ではない。ため、代入の左辺におけない。
この非対称性が、「配列とポインタは同じ」と言う誤解を生みやすい原因の一つ。
