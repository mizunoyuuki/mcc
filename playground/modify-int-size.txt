現状int, int *のみの型を実装しており、これをint 4バイト、int * 8バイトのように、異なるバイト数でスタックを管理できるように実装を進める。

現在、int, char など固定配列で管理しているため、後々typedefで動的に型名が追加されたりすると、うまく動作しなくなる。
そこで、動的に管理できる仕組みを追加し、int を4バイトで処理できるようにする。

その上で、いくつかのフェーズに分けてインクリメンタルに実装できるような方法で進める.

まず現状の整理。

TypeSpecifier, Typeと言う二つの構造体を使って管理しているが、今後のtypedefを見据えてどう実装すればいいのか。

TypeSpecifier

typedef struct TypeSpecifier {
  char *type_name;        // "int", "char"とかの文字を保存している。
  TokenKind token_kind;   // 列挙子 TK_INT_TYPE, TK_CHAR_TYPE
  int len;                // 文字の長さ(int 3, char 4)
  int size;               // バイトサイズ
}

typedefで新しく定義した型を追加できるようにするため、
TypeRegister型を作成。

mainの最初にint, charなどの言語レベルでサポートしている型の連結リストを作っておく.

int: 4バイト
char: 1バイト
ptr : 8バイト

と8バイト以外のメモリアクセスをしようとすると、generatorで出力する命令も変更する必要がある。


命令を変更する必要があるって箇所の調査

x86のメモリアクセス幅と、命令の対応。
- x86-64では同じ「メモリから読む」でもサイズごとに異なるオペランドが必要を指定が必要。

mov rax, [addr]
は、8バイト

mov eax, [addr]
は4バイト(上位をゼロクリア)

mov al, [addr]
は１バイト

Intel構文では、dword ptr, qword ptr, byte ptrサイズを明示できる.

AT&T構文(GASではデフォルト)では、命令にサフィックスをつけてサイズをしめす。
movq, movl, movw, movb

Intel構文のdword ptrよりこちらの方が簡潔だが、読みにくいと言う人も多い。linuxカーネルもIntel構文への移行が議論されたこともある。

このmcc実装で、Nodeにもtype構造体を紐づけている理由は、アセンブリ生成時に、その変数が4バイトアクセスか、8バイトアクセスか、1バイトアクセスかで出力する命令が、例えば同じ足し算でも変わるため。

この辺りのバイト数の違いによってアセンブリの学習になるね。
